## 1. Источник и состав данных

### Используемые датасеты

| Датасет | Источник | Объём | Формат | Роль |
|--------|---------|-------|--------|------|
| `Romjiik/Russian_bank_reviews` | Hugging Face | ~200 000 отзывов | text, rating (1–5), date | Основной: рейтинг ≤ 2 → `critical` |

### Целевая переменная

Бинарная метка: `critical` (1) / `not_critical` (0).

Правила формирования метки:
- `Russian_bank_reviews`: `rating ≤ 2` → `critical = 1`
- `Toxic_Russian_Comments`: `label == "THREAT"` → `critical = 1`; остальные → `critical = 0`
- `Russian_Inappropriate_Messages`: темы экологии, здоровья, насилия, мошенничества → `critical = 1`; нейтральные → `critical = 0`

### Поля датасета после объединения

| `date` | datetime | Дата публикации (только для bank_reviews) |

## 2. Базовый EDA с выводами для моделирования

### 2.1 Распределение классов

**Оценочное распределение после объединения:**

| Класс | Примерное количество | Доля |
|-------|---------------------|------|
| `not_critical` (0) | ~180 000 | ~83% |
| `critical` (1) | ~36 000 | ~17% |
| **Итого** | **~216 000** | **100%** |

> Дисбаланс ~1:5 — умеренный, но значимый для задачи.

**Вывод для моделирования:** использовать `class_weight='balanced'` или взвешенный loss. Основная метрика — Recall по классу `critical` и PR-AUC, а не Accuracy (при дисбалансе Accuracy вводит в заблуждение).

### 2.2 Длина текстов

**Оценочные характеристики:**

| Датасет | Медиана токенов | Макс. токенов | Характер |
|--------|----------------|--------------|---------|
| bank_reviews | ~80 | ~500 | Развёрнутые отзывы |
| toxic_comments | ~15 | ~100 | Короткие комментарии |
| inappropriate_msgs | ~25 | ~150 | Средние сообщения |

**Вывод для моделирования:** RuBERT поддерживает до 512 токенов. Большинство текстов укладываются в 128 токенов — можно использовать `max_length=128` для ускорения обучения без потери информации. Для длинных отзывов (bank_reviews >200 токенов) применять `truncation=True`.

### 2.3 Распределение по времени (bank_reviews)

**Характеристики:** данные собраны за период ~2018–2023. Количество отзывов стабильно до 2020 года, затем растёт с появлением цифрового банкинга.

**Вывод для моделирования:** временно́е распределение данных позволяет применить `TimeSeriesSplit` для честной валидации. Тестовая выборка должна содержать только посты, датированные позже обучающей — это имитирует реальную работу системы.

### 2.4 Анализ токенов и лексики

**Характерные токены в классе `critical` (оценочно):**
- Финансовая сфера: *списали, обман, мошенники, заблокировали, украли*
- Угрозы/токсичность: *угрожает, убью, убьют, опасность*
- ESG-лексика (целевая): *разлив, выброс, загрязнение, химикаты, авария, эвакуация*

**Важный вывод:** Лексика bank_reviews и toxic_comments **слабо пересекается** с целевой ESG-лексикой. Это подтверждает главный риск проекта: **доменное смещение**. Модель, обученная только на этих данных, может не распознать «разлив нефти в реке» как критический инцидент.

**Решение:** дополнить датасет LLM-предразметкой реальных Telegram-постов с ESG-тематикой (хотя бы 500–1000 примеров).

## 3. Оценка качества разметки и предложения по улучшению

### 3.1 Текущее качество разметки

**`Russian_bank_reviews`** — разметка косвенная (по рейтингу):
- ✅ Сильная сторона: нет человеческого субъективизма, рейтинг — однозначный сигнал
- ⚠️ Слабая сторона: рейтинг 1–2 ≠ критический инцидент автоматически. Примеры ложных меток:
  - «Очень долгое ожидание в очереди» (рейтинг 1) → не критично для ESG
  - «Приложение упало» (рейтинг 2) → не критично
- **Оценка точности метки:** ~70–75% корректных примеров для задачи EcoPulse

**`Toxic_Russian_Comments`** — ручная разметка краудсорсингом:
- ✅ Чёткие классы, профессиональная разметка
- ⚠️ Класс THREAT слишком широкий: включает личные угрозы между пользователями, которые не являются ESG-инцидентами
- **Оценка точности метки:** ~60% корректных примеров для задачи EcoPulse

**`Russian_Inappropriate_Messages`** — академическая разметка:
- ✅ 18 тематических категорий, детальная схема
- ✅ Публикация EACL 2021 — высокое качество
- **Оценка точности метки:** ~80% корректных примеров для задачи EcoPulse

### 3.2 Предложения по улучшению разметки

1. **LLM-предразметка с ESG-промптом** — прогнать все три датасета через GPT-4o с промптом эксперта по репутационным рискам. Это даст дополнительный слой фильтрации и уберёт явные ложные метки.

2. **Ручная проверка пограничных случаев** — отобрать посты с уверенностью модели 0.45–0.55 (неопределённая зона) и разметить вручную вместе с ESG-аналитиком заказчика.

3. **Добавить реальные ESG-тексты** — 500–1000 постов из открытых Telegram-каналов с ручной разметкой. Это повысит репрезентативность выборки для целевого домена.

4. **Межразметчиковое согласие (IAA)** — для новой разметки использовать двух разметчиков и мерить Cohen's Kappa. Допустимый порог: κ ≥ 0.7.

## 4. Алгоритм формирования выборки и стратегия валидации

### 4.1 Алгоритм формирования финального датасета

```
Шаг 1. Загрузка трёх датасетов
        ↓
Шаг 2. Перемаркировка под единую схему critical / not_critical
        (правила по п. 1)
        ↓
Шаг 3. Очистка текстов
        - удаление дубликатов (deduplication по text)
        - удаление пустых и слишком коротких текстов (< 5 токенов)
        - нормализация: нижний регистр, удаление HTML-тегов
        ↓
Шаг 4. LLM-предразметка сомнительных примеров
        - прогнать пограничные случаи через GPT-4o
        - отфильтровать явно нерелевантные для ESG примеры
        ↓
Шаг 5. Добавление реальных Telegram-постов
        - 500–1000 вручную размеченных постов из открытых каналов
        ↓
Шаг 6. Финальный датасет: ~50 000–70 000 примеров
        (после очистки и фильтрации)
```

### 4.2 Стратегия валидации

**Используем временну́ю валидацию (Time-based split)** — единственная честная стратегия для задачи мониторинга, где модель должна работать на новых данных.

#### Разбивка датасета

| Выборка | Данные | Размер | Назначение |
|---------|--------|--------|-----------|
| Train | bank_reviews до 2021 + все остальные источники | ~70% | Обучение модели |
| Validation | bank_reviews 2021–2022 | ~15% | Подбор гиперпараметров, ранняя остановка |
| Test | bank_reviews 2023 + новые Telegram-посты | ~15% | Финальная оценка (только один раз!) |

> Для датасетов без даты (toxic_comments, inappropriate_msgs) — случайное разбиение внутри страты, но только в Train/Val. В Test кладём только датированные данные.

#### Дополнительно: кросс-валидация с `TimeSeriesSplit`

При подборе гиперпараметров применять `TimeSeriesSplit(n_splits=5)` на обучающей выборке:

```
Fold 1: Train [t0 → t1] | Val [t1 → t2]
Fold 2: Train [t0 → t2] | Val [t2 → t3]
Fold 3: Train [t0 → t3] | Val [t3 → t4]
...
```

Это имитирует реальный сценарий: модель всегда обучается на прошлом и предсказывает будущее.

#### Стратификация

При разбивке соблюдать пропорцию классов в каждой выборке (stratified split по `label`), чтобы редкий класс `critical` не оказался недопредставлен в Val и Test.

### 4.3 Baseline для сравнения

Перед обучением RuBERT построить два baseline:

| Модель | Ожидаемый F1 (critical) | Зачем |
|--------|------------------------|-------|
| TF-IDF + LogisticRegression | ~0.65–0.70 | Быстрый нижний ориентир |
| TF-IDF + CatBoost | ~0.70–0.75 | Более сильный baseline |

RuBERT обязан превзойти оба baseline, иначе fine-tuning не оправдан.
