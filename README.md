# EcoPulse — Система мониторинга ESG-репутации

> Система раннего обнаружения экологических и социальных скандалов, связанных с крупными компаниями, на основе анализа локальных Telegram-каналов и региональных медиа.

---

## Идея

Крупные компании — промышленные холдинги, ретейл, энергетика — каждый день упоминаются в тысячах постов в локальных Telegram-каналах, городских пабликах и региональных медиа. Большинство этих упоминаний — фоновый шум. Но иногда среди них появляется что-то по-настоящему опасное: жалоба на разлив химикатов, пост о нарушении прав работников, сигнал о массовом обмане клиентов.

Если PR-команда узнаёт об этом **через три дня** - скандал уже разгорелся.
Если **через три часа** - его можно погасить в зародыше.

**EcoPulse** решает именно эту задачу: непрерывно сканирует поток текстов и мгновенно сигнализирует, когда в нём появляется реальная репутационная угроза.

---

## Почему это ценно

Корпоративный PR - миллиардный рынок. Компании нанимают целые отделы мониторинга, которые вручную просматривают публикации. EcoPulse автоматизирует эту рутину и даёт конкурентное преимущество: **скорость реакции измеряется минутами, а не сутками**.

Ключевая бизнес-метрика - **Time-to-Alert**: чем быстрее компания узнала об инциденте, тем меньше репутационный и финансовый ущерб.

---

## ML-задачи

Три слоя анализа работают последовательно:

### 1. Бинарная классификация (основная задача)
Пост либо содержит критический инцидент (`critical_incident`), либо нет (`not_critical`).
Высокий **Recall** по классу «критично» — чтобы не пропустить ни одной реальной угрозы.
Высокий **Precision** - чтобы не заваливать аналитика ложными тревогами.

### 2. NER (Named Entity Recognition)
Из критического поста извлекаются конкретные сущности:
- название компании / бренда
- географический объект (регион, завод, адрес)
- тип инцидента (экология, здоровье, финансы)

Позволяет маршрутизировать уведомление нужному отделу.

### 3. Stance Detection
Определяем, является ли пост жалобой / протестом или просто нейтральным упоминанием.
Один и тот же текст с упоминанием экологии может быть **пресс-релизом компании** или **криком жителей** - это принципиально разные случаи.

## Датасеты

### 1. `Romjiik/Russian_bank_reviews` - **основной**
🔗 https://huggingface.co/datasets/Romjiik/Russian_bank_reviews

Отзывы клиентов российских банков с числовым рейтингом (1–5) и датой.

**Чем полезен для проекта:**
Отзывы описывают конкретные проблемы - несанкционированные списания, обман, игнорирование жалоб. Это и есть «критические инциденты» в финансовой сфере. Метку `critical` можно получить автоматически из поля `rating`, **без ручной разметки** - это ускоряет старт в разы. Поле `date` позволяет использовать временну́ю валидацию (`TimeSeriesSplit`), что критично для реальной задачи мониторинга.
Также есть положительные отзывы.

---

### 2. `AlexSham/Toxic_Russian_Comments` - **расширение домена**
🔗 https://huggingface.co/datasets/AlexSham/Toxic_Russian_Comments

Размеченные комментарии из российской соцсети ok.ru.
Классы: `NORMAL`, `INSULT`, `THREAT`, `OBSCENITY`.

**Чем полезен для проекта:**
Класс `THREAT` - это сигналы прямой угрозы или агрессии. В контексте EcoPulse такие сообщения напрямую соответствуют «критическому инциденту»: угрозы в адрес компании или её сотрудников требуют немедленной реакции. Добавление этого датасета расширяет тематику за пределы банков и учит модель распознавать агрессию в разговорном, неформальном тексте - именно таком, который пишут в Telegram.

---

## Стратегия сборки финального датасета

```
Romjiik/Russian_bank_reviews  (rating ≤ 2 → critical)
        +
AlexSham/Toxic_Russian_Comments  (THREAT → critical)
        +
  [еще датасеты будут]
        ↓
  Единая схема: { text, label: critical | not_critical, date }
        +
  LLM-предразметка свежих постов
  с выборочной ручной проверкой сомнительных случаев
```

Итог: **несколько тысяч размеченных примеров** без ручного труда + возможность наращивать выборку новыми доменами.

## Архитектура решения

```
[Telegram / RSS] → [Сбор постов] → [RuBERT classifier]
                                          ↓
                               critical_incident?
                              ↙               ↘
                           ДА                 НЕТ
                            ↓
                    [NER: компания + место]
                            ↓
                   [Telegram-уведомление]
                   [Дашборд Streamlit]
```

## Метрики успеха

| Метрика | Целевое значение | Зачем |
|--------|-----------------|-------|
| Recall (critical) | ≥ 0.85 | Не пропустить реальный инцидент |
| Precision (critical) | ≥ 0.75 | Не заваливать аналитика ложными тревогами |
| F1 | ≈ 0.80 | Общий баланс |
| Time-to-Alert | < 15 мин | Бизнес-KPI |
