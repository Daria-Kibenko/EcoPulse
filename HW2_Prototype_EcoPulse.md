# Домашнее задание №2 - Прототип продукта

**Проект:** EcoPulse - Система мониторинга ESG-репутации  
**Этап:** Уточнение прототипа продукта

---

## 1. Ключевая гипотеза

> **Формат:** Если мы предлагаем продукт A для сегмента B в сценарии C, то мы получим метрику D не меньше E в течение времени T, потому что [причина].

**Гипотеза:**

Если мы предложим **Telegram-бот с автоматической классификацией ESG-инцидентов на базе RuBERT** для **ESG-аналитиков и PR-менеджеров крупных промышленных компаний** в сценарии **ежедневного мониторинга локальных Telegram-каналов**, то мы получим **Time-to-Alert не более 4 часов** в течение **первых 4 недель тестирования**, потому что **модель непрерывно сканирует поток постов и мгновенно уведомляет о критических инцидентах, устраняя ручной поиск и задержку в цепочке эскалации**.

---

**Один сегмент:** ESG-аналитик / PR-менеджер промышленной компании (нефтегаз, металлургия, химия) - человек, отвечающий за мониторинг репутационных рисков и подготовку реакции на инциденты.

**Один сценарий:** Аналитик начинает рабочий день и вместо ручного просмотра 30-50 каналов открывает дайджест от бота - уже с расставленными приоритетами (🔴 критично / 🟡 средний риск / ⚪ фон). Если критический пост появился ночью - бот уже отправил уведомление.

**Один способ монетизации:** B2B SaaS - ежемесячная подписка за одну компанию-заказчика. Ценовая логика: стоимость одного предотвращённого репутационного скандала в разы превышает стоимость подписки.

## 2. Ключевые метрики

### Продуктовые метрики

| Метрика | Baseline | Цель | Способ измерения | Частота |
|--------|---------|------|-----------------|---------|
| Time-to-Alert (медиана) | ~48 ч (ручной мониторинг) | ≤ 4 ч | Лог: время публикации поста vs время уведомления бота | Еженедельно |
| Recall на реальных постах | — | ≥ 85% | Сравнение предсказаний модели с фидбеком аналитика (👍/👎) | По итогам теста |
| Доля ложных тревог (1 − Precision) | — | ≤ 25% | Фидбек 👎 / все уведомления | Еженедельно |
| Retention аналитика | — | ≥ 70% на 14-й день | Лог активности Telegram-бота | Еженедельно |

### Технические метрики

| Метрика | Baseline | Цель | Способ измерения | Частота |
|--------|---------|------|-----------------|---------|
| Recall (критический + средний, macro) | TF-IDF + LR: ~0.70 | ≥ 0.85 | sklearn, hold-out с временно́й валидацией | После каждого эксперимента |
| Precision (те же классы) | ~0.65 | ≥ 0.75 | Там же | После каждого эксперимента |
| F1 (0.8 × Recall + 0.2 × Precision) | ~0.68 | ≥ 0.83 | Там же | После каждого эксперимента |
| Latency инференса (1 пост) | — | ≤ 200 мс | time.time() на ONNX-модели | При деплое |
| Uptime бота | — | ≥ 99% | UptimeRobot | Непрерывно |

## 3. Что входит в прототип / что намеренно не делаем

### Входит в прототип

- **Парсер** открытых Telegram-каналов (список из 20-30 каналов) через Telethon - раз в 15 минут
- **Бинарная классификация** постов: `critical` / `not_critical` на базе дообученного RuBERT (упрощение от 4-классовой схемы - ради скорости)
- **Telegram-бот**: уведомление при обнаружении критического поста + утренний дайджест топ-5 постов по скорингу
- **Кнопка фидбека** в боте (👍 / 👎) - сбор живых меток для улучшения модели
- **Дашборд** (Streamlit): список постов за день с цветовым приоритетом и ссылками

### Намеренно не делаем (и почему)

| Что не делаем | Причина |
|--------------|---------|
| NER (компании, геолокации) | Требует отдельного обучения; для проверки гипотезы не нужен — аналитик сам читает пост |
| Stance Detection | Усложняет разметку; оставляем на вторую итерацию |
| 4-классовая схема | Бинарная достаточна для валидации гипотезы; размечается быстрее |
| Интеграция с CRM / Slack | Требует участия IT заказчика; для MVP избыточно |
| Мониторинг закрытых каналов | Юридические и технические риски |
| Авто-ретрейн модели | Не нужен на этапе прототипа; модель статична |

## 4. Готовые блоки vs самодел

### Берём готовое

| Инструмент | Что даёт | Почему готовое                                  |
|-----------|---------|-------------------------------------------------|
| `ai-forever/ruBert-base` (HF) | Предобученная модель для русского текста | Дообучить - неделя; с нуля - месяцы и терабайты |
| `Romjiik/Russian_bank_reviews` | Размеченные отзывы, старт обучения без ручного труда | Тысячи примеров бесплатно                       |
| `AlexSham/Toxic_Russian_Comments` | Расширение выборки токсичными комментариями | Уже размечено, бесплатно                        |
| Telethon | Парсинг Telegram | Зрелая библиотека, поддерживает публичный API   |
| python-telegram-bot | Уведомления и кнопки фидбека | Стандарт для Telegram-ботов                     |
| Streamlit | Дашборд | Готовый UI за 50 строк кода                     |
| ONNX Runtime | Быстрый инференс | Квантизация без потери качества                 |

### 🛠 Делаем сами (и почему это критично)

| Компонент | Почему делаем сами                                                                    |
|----------|---------------------------------------------------------------------------------------|
| **Fine-tuning RuBERT** под задачу EcoPulse | Готовых моделей для ESG-инцидентов на русском не существует - это ключевое ноу-хау    |
| **Схема разметки и LLM-промпт** для предразметки | Определение «критического инцидента» - бизнес-экспертиза, нельзя делегировать         |
| **Скоринг угрозы** (вероятность как числовой сигнал) | Готовые модели дают только класс; непрерывный скор нужен для ранжирования в дайджесте |
| **ETL-пайплайн**: парсинг → очистка → инференс → уведомление | Специфичен под архитектуру; стандартных решений нет                                   |

## 5. Главный риск и формат прототипа

**Главный риск:** Модель, обученная на отзывах о банках и токсичных комментариях, не распознаёт ESG-инциденты в Telegram - стиль, лексика и контекст принципиально отличаются.

**Почему это главный риск:** Если Recall < 0.60 на реальных постах - продукт не работает совсем. Это убивает гипотезу раньше, чем мы успеем что-либо улучшить.

**Как проверяем прототипом:** Запускаем бот на 2–3 реальных канала на 2 недели, сравниваем предсказания с ручной оценкой аналитика через кнопку 👍/👎.

---

### Формат прототипа

```
[Telegram-каналы]
       │
  (Telethon, раз в 15 мин)
       │
[Парсер + очистка текста]
       │
[RuBERT классификатор]
  critical / not_critical
       │
  ┌────┴────────────────┐
critical           not_critical
  │                  (в лог)
[Telegram-бот]
  → уведомление PR-менеджеру
  → утренний дайджест топ-5
  → кнопка 👍/👎

[Streamlit дашборд]
  → все посты за день
  → цветовые приоритеты
  → ссылки на источники
```

**Технический стек:** Python 3.11, Telethon, HuggingFace Transformers, ONNX Runtime, python-telegram-bot, Streamlit, SQLite, деплой на VPS (4 ГБ RAM, Ubuntu).

## 6. Критерии остановки теста прототипа

### Досрочная остановка - при выполнении любого условия:

| Условие | Порог | Интерпретация |
|---------|-------|--------------|
| Recall на реальных постах | < 0.60 за первые 2 недели | Модель не работает на домене; нужна переразметка или смена архитектуры |
| Доля ложных тревог | > 50% за неделю | Аналитик перестаёт доверять боту |
| Retention аналитика | < 40% на 7-й день | Продукт не встраивается в рабочий процесс |
| Uptime бота | < 90% за неделю | Инфраструктура нестабильна |

### Критерии успешного завершения (переход к следующей итерации):

- Recall ≥ 0.80 на реальных постах (по фидбеку 👍/👎)
- Доля ложных тревог ≤ 30%
- Retention ≥ 60% на 14-й день
- Хотя бы 1 реальный критический инцидент обнаружен раньше, чем аналитик нашёл его вручную
