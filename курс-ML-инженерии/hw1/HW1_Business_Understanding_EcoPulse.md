# Домашнее задание №1 - Бизнес-анализ (Business Understanding)

**Проект:** EcoPulse - Система мониторинга ESG-репутации  
**Методология:** CRISP-DM

---

## 1. Бизнес-анализ (Business Understanding)

### 1.1 Проблема бизнеса и бизнес-цель

**Контекст:** Крупная промышленная корпорация регулярно сталкивается с локальными экологическими и социальными протестами - загрязнение воздуха, разливы, конфликты с местными сообществами. Информация об инцидентах появляется в региональных Telegram-каналах и небольших новостных пабликах за несколько дней до выхода в федеральные СМИ. Ручной мониторинг (Google Alerts, подписка на ограниченный набор каналов) не успевает отслеживать тысячи источников, из-за чего компания упускает момент для упреждающей реакции.

**Бизнес-цель:** Снизить репутационные и финансовые риски путём сокращения времени реагирования (Time-to-Alert) на ESG-инциденты с 2-3 суток до 2-4 часов. Это позволит PR- и GR-департаментам своевременно готовить комментарии, смягчать последствия и не допускать эскалации в федеральную повестку.

---

### 1.2 Заинтересованные стороны и принятие решений

- **Спонсор проекта:** Директор по устойчивому развитию (ESG-директор), бюджет утверждается на уровне правления
- **Ключевые ЛПР:** Начальник департамента по связям с общественностью и GR-директор
- **Основные пользователи:** Менеджеры по устойчивому развитию, специалисты PR и Government Relations (GR)
- **Эксперты от заказчика:** Два ESG-аналитика для разметки данных, валидации отчётов и обратной связи
- **Куратор от заказчика:** Руководитель отдела риск-менеджмента (принимает промежуточные результаты, согласует метрики)

---

### 1.3 Существующие решения и их ограничения

| Решение | Недостаток | Что закрывает EcoPulse |
|--------|-----------|----------------------|
| Ручной мониторинг | Медленно, дорого, не масштабируется | Автоматическое обнаружение за ≤ 4 часов |
| Brand Analytics, YouScan, Медиалогия | Плохо покрывают локальные Telegram-каналы с аудиторией 500–2000 человек; нет ESG-специфики | Фокус именно на локальных и нишевых источниках |
| Общий sentiment analysis | Нет скоринга серьёзности; не адаптирован под бизнес-процессы заказчика | Точная многоклассовая классификация ESG-инцидентов с оценкой угрозы |

---

### 1.4 Ключевые бизнес-переменные

Модель прогнозирует **«критичность инцидента»** - составной показатель, определяющий приоритетность реакции:

- **Факт обнаружения ESG-инцидента** в источнике (бинарно)
- **Стадия развития:** «ранний сигнал» / «активное обсуждение» / «вирусное распространение»
- **Степень угрозы для репутации:** низкая / средняя / высокая - оценивается на основе тональности, авторитетности источника и скорости роста вовлечённости

На основе прогноза система формирует уведомление с категорией срочности, что напрямую влияет на целевую бизнес-переменную: **медианное время реакции на критический инцидент**.

---

### 1.5 SMART-метрики успеха проекта (бизнес-уровень)

| Критерий | Описание |
|---------|---------|
| **Specific** | Сократить медианное время от публикации первого сообщения об инциденте в локальном Telegram-канале до его регистрации в дашборде аналитика и отправки уведомления |
| **Measurable** | Время измеряется в часах; фиксируется логами системы и данными о действиях аналитика |
| **Achievable** | По пилотным замерам ручного мониторинга текущее медианное время ~48 часов; автоматизация позволяет довести до 3–4 часов |
| **Relevant** | Напрямую влияет на KPI PR- и GR-департаментов: количество предотвращённых «вирусных» скандалов и снижение негативных упоминаний в федеральных СМИ |
| **Time-bound** | Достичь целевого сокращения через 3 месяца после ввода в эксплуатацию, но не позднее чем через 6 месяцев после окончания разработки |

### 1.6 Ресурсы

#### 1.6.1 Оборудование и инфраструктура

- **Облако** (Yandex Cloud / SberCloud): ВМ для парсинга (8 vCPU, 32 ГБ ОЗУ), СУБД для хранения данных
- **Среда экспериментов:** JupyterHub на выделенном сервере

#### 1.6.2 Данные и доступ

- **Внутренние данные:** Заказчик предоставляет исторические журналы уже обработанных вручную инцидентов (глубина 2 года, ~5 000 записей) - основа для разметки и обучения
- **Внешние данные:** Публичные API Telegram, RSS-ленты региональных СМИ; доступ не требует специальных разрешений при соблюдении ToS
- **Дополнительные данные:** При необходимости закупаются выгрузки из платных систем мониторинга для расширения обучающей выборки (заложено в бюджет)

#### 1.6.3 Экспертиза заказчика

Два ESG-аналитика на **25% рабочего времени** в течение ~3 месяцев активной разметки и валидации. Они также участвуют в составлении правил классификации и согласовании итоговых отчётов.

### 1.7 Риски и план митигации

| Риск | Вероятность | Влияние | Митигация |
|------|------------|---------|-----------|
| Дисбаланс классов (мало критических инцидентов) | Высокая | Высокое | Взвешенный loss, аугментация (back-translation, synonyms), PR-AUC вместо Accuracy |
| Высокий уровень шума (фейки, нерелевант) | Высокая | Среднее | Быстрый фильтр-классификатор релевантности перед глубоким анализом; ручная верификация первой партии для калибровки порога |
| Смещение домена: отзывы → Telegram | Средняя | Высокое | LLM-предразметка реальных Telegram-постов; few-shot learning + синтетическая генерация |
| Отсутствие стабильных закономерностей | Низкая | Высокое | Пилотный анализ; при провале — документирование и закрытие на стадии Data Understanding |
| Задержки в поставке экспертной разметки | Средняя | Среднее | Подробный гайд по разметке, два эксперта с перекрёстной проверкой, промежуточные контрольные точки |
| Дрейф данных со временем | Средняя | Среднее | Ежемесячный ретрейн, мониторинг Recall в продакшне |
| Техническая сложность интеграции | Средняя | Среднее | Конвейер проектируется независимым; экспорт уведомлений через Webhook / Telegram Bot API |
| Не уложиться в сроки | Низкая | Среднее | Разбить на MVP (классификация) и расширение (NER, Stance); буфер 1 неделя |

### 1.8 Постановка задачи в терминах машинного обучения

**Задача:** Многоклассовая классификация коротких текстов с дополнительным скорингом серьёзности.

**Входные данные:** текст сообщения + метаданные (размер канала, частота постинга, время публикации, геотег при наличии)

**Целевая переменная:** категория инцидента - `Неинцидент` / `Низкий риск` / `Средний риск` / `Критический риск` - с учётом стадии развития

**Подход:** Дообучение предобученной языковой модели (RuBERT / ruRoBERTa) на классификацию + дополнительный регрессионный выход для скоринга угрозы. Для учёта динамики обсуждения возможно применение последовательных моделей на агрегированных данных за час.

**Метрики модели (офлайн-оценка):**

| Метрика | Роль | Целевое значение |
|---------|------|-----------------|
| Recall (Средний + Критический, macro) | Основная - нельзя пропустить угрозу | ≥ 0.92 |
| Precision (те же классы) | Вспомогательная - контроль ложных срабатываний | ≥ 0.75 |
| F1 (0.8 × Recall + 0.2 × Precision) | Итоговая - отражает бизнес-приоритет полноты | ≥ 0.83 |
| PR-AUC | При сильном дисбалансе классов | ≥ 0.80 |

---

### 1.9 Критерии успешности модели

- **Минимально допустимый уровень:** Recall для критических инцидентов ≥ 0.85. Если ниже - модель не внедряется, требуется дообучение или пересмотр подхода
- **Целевой (оптимальный) уровень:** Recall ≥ 0.92 при Precision ≥ 0.75 (F1 ≥ 0.83)
- **Экспертная приёмка:** Слепое тестирование - 100 произвольных сообщений за последний месяц, не попавших в обучающую выборку, оцениваются ESG-аналитиками заказчика. Допустимое расхождение с предсказаниями модели — не более 15%

## Технологический стек

- **Модель:** `ai-forever/ruBert-base` или `DeepPavlov/rubert-base-cased`
- **Обучение:** Hugging Face `transformers` + `Trainer`
- **Валидация:** `TimeSeriesSplit` по дате публикации
- **Инфраструктура:** Yandex Cloud / SberCloud, Tesla V100/A100
- **Интерфейс:** Streamlit / Gradio
- **Бот:** Telethon + python-telegram-bot, webhook в Slack
- **Инференс:** ONNX-квантизация для скорости
